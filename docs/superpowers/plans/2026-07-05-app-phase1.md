# DeepVerse App — Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan milestone-by-milestone. This plan is milestone/task structured; within each task, work TDD (write the test, watch it fail, implement, pass, commit). Expand a task into finer steps if it's doing too much.

**Goal:** Build the DeepVerse client-side PWA — a Home page and a split-screen Study mode with the interpretive-difference workbench — entirely on top of `bible.db` v2. Data-driven, **no AI**, Scholarly-print light/dark.

**Architecture:** A static single-page app. `bible.db` (built by Plan 1) ships as a static asset; **sql.js** (WASM SQLite) loads it into memory in the browser and every feature is a SQL query against it. Notes / prefs / reading-activity / to-study live in **IndexedDB + localStorage**. A service worker caches the shell + DB for offline. No backend, no auth.

**Tech Stack:** Vite + **Svelte** (lightweight, reactive — React is an acceptable swap; the data layer is framework-agnostic), `sql.js`, `idb` (thin IndexedDB wrapper), `vite-plugin-pwa` (service worker). Vitest + @testing-library for tests.

## Global Constraints

- **Fully client-side, local-first, no auth, no backend.** (Spec §2.)
- **Scholarly-print theme only**, light + dark, icon-only ☾/☀ toggle, OS-default + manual override + persisted. Dark = warm candlelit (not cold slate). Palette/type are in the mockups & spec §5. Avoid shadcn/Tailwind defaults and the cream+display-serif+sage "tasteful default" (unslop-ui).
- **A = oxblood, B = indigo** everywhere; "both" = double underline. Difference presence is **binary** (A / B / both / nothing) — never a "none" category.
- **Buttons imply a choice:** Type A = comparison buttons; Type B = inline underlined word (spec §8 affordance rule).
- Original-language words show **transliteration/pronunciation**.
- Ship `bible.db` as a static asset loaded once via `sql.js`; cache via service worker; queries run in-browser.
- Reading-activity is **auto-tracked** (opening a chapter counts a day).

## Milestone 1 — Data access layer (`src/lib/db.js`)

**Deliverable:** a framework-agnostic module with a tested function per feature. This is the spine — get it right and the UI is thin. All functions are synchronous after `await loadDb()` (sql.js is in-memory).

### Task 1.1 — Loader + `query`

**Interfaces:** `export async function loadDb(url='/bible.db'): Promise<void>`; `export function query(sql, params=[]): any[]` (throws if not loaded).

```js
// src/lib/db.js
import initSqlJs from 'sql.js';
let db = null;
export async function loadDb(url = '/bible.db') {
  const SQL = await initSqlJs({ locateFile: () => '/sql-wasm.wasm' });
  const buf = await (await fetch(url)).arrayBuffer();
  db = new SQL.Database(new Uint8Array(buf));
}
export function query(sql, params = []) {
  const stmt = db.prepare(sql); stmt.bind(params);
  const rows = []; while (stmt.step()) rows.push(stmt.getAsObject());
  stmt.free(); return rows;
}
```
Test: after loading the real `public/bible.db`, `query('SELECT COUNT(*) n FROM verses')[0].n > 90000`.

### Task 1.2 — Reader + versions

**Interfaces:**
- `getChapter(version, book, chapter): {verse:number, text:string}[]`
- `getVerseAllVersions(book, chapter, verse): {version, text}[]`
- `listBooks(): {book, name, chapters:number}[]` (from a static OSIS table in `refs.js` + `verses`)
- `getChapterLanguages(book, chapter): ('grc'|'hbo'|'arc')[]` →
  `query("SELECT DISTINCT lang FROM words WHERE book=? AND chapter=?", [book,chapter]).map(r=>r.lang)`

Tests: `getChapter('NIV','John',3)` returns 36 rows; `getChapterLanguages('Dan',2)` returns `['hbo','arc']` (order-insensitive); `getVerseAllVersions('John',3,16)` returns 3 rows.

### Task 1.3 — Interlinear (Original card)

**Interfaces:** `getInterlinear(book, chapter, verse): {position, original, translit, gloss, strongs, morph, lemma}[]` ordered by position. Plus `getLexicon(strongs): {lemma, translit, gloss, definition} | null` (try exact `code`, then base without trailing letter).

Test: `getInterlinear('John',3,16)[0].translit === 'houtōs'`; `getLexicon('G0025').definition` contains "love".

### Task 1.4 — Differences (the engine's read side)

**Interfaces:**
- `getVerseDifferences(book, chapter, verse): {position, type:'A'|'B', strongs, detail:object, original, translit, gloss}[]`
  — join `differences` to `words` on (book,chapter,verse,position) for the display word; `JSON.parse(detail)`.
- `getChapterDifferenceMap(version, book, chapter): Map<verse, {A:number[], B:number[]}>` — which word
  positions in each verse are A/B, to drive the in-text underlines. (For the reader we only have English
  text, not word positions; underline by matching the difference word's English gloss occurrence in the
  verse text — see note.) Return per-verse the set of English tokens to underline and their type.
- For **Type A** detail, resolve the near-synonym display: for each `nearSynonyms[i].strongs`, fetch its
  lemma+translit via `getLexicon`.

**Underline-mapping note (important, spec §7):** the reader shows the *English* verse; the difference
engine keys off *original-word* positions. To underline the English word, map via the interlinear gloss:
for each differenced position, take `words.gloss` (e.g. "loves"/"life"), find that word in the English verse
text (case-insensitive, first unmatched occurrence), and wrap it. This is approximate (no NIV↔Greek
alignment exists — spec §13); acceptable for Phase 1. Implement as `underlineSpans(englishText, diffs)`
returning `[{text, type|null}]` segments for the renderer.

Tests: `getVerseDifferences('John',21,15)` includes a Type A on the ἀγαπάω word; `getVerseDifferences('John',12,25)` includes a Type B (ψυχή) and a Type A (φιλέω); `underlineSpans('Anyone who loves their life...', diffs)` marks "loves" (A) and "life" (B).

### Task 1.5 — Cross-references + Context

**Interfaces:** `getCrossRefs(book, chapter, verse): {to_ref, votes}[]` ordered by votes desc; `getChapterCrossRefStats(book, chapter)` (counts). Test: `getCrossRefs('John',3,16)[0].to_ref === 'Rom.5.8'`.

### Task 1.6 — Stats + word-selector concordance

**Interfaces:**
- `countEnglishWord(version, word): number` — occurrences of an English word across the whole version
  (`SELECT text FROM verses WHERE version=?` then count regex matches — or a LIKE prefilter then exact
  word-boundary count in JS).
- `countLemma(strongs): {total:number, byBook:{book,n}[]}` — concordance of an original lemma across the
  whole Bible from `words`.
- `verseWordCounts(version, book, chapter, verse)` — word/char counts.

Tests: `countLemma('G0025').total` ≈ 143 (agapaō); `countEnglishWord('NIV','love') > 300`.

Commit each task. **This milestone is the highest-value; write thorough tests here.**
